# Smart NPCs — Design Spec

- **Date:** 2026-06-27
- **Project:** gs26-smart-npc
- **Status:** Brainstormed, pending user spec review
- **Context:** Sony / PlayStation internal hackathon. Browser-based demo prototype.

## 1. Problem & Vision

Non-playable characters (NPCs) in modern games are pre-scripted: their dialogue
is fixed per scenario and they have no memory of what the player has done outside
their immediate encounter. The vision is NPCs with a shared awareness of the
player's actions across the game world, propagated through an in-world social
medium, with dialogue and behavior dynamically generated by a large language
model (LLM).

Example payoff: in one scene the player saves a girl from a fire; several
scenes later, the player enters a restaurant in another district and the owner
refuses payment because the rescued girl was his niece — the news reached him
through the in-world social network and he references the specific deed in
unscripted natural language.

## 2. Demo Goals (the three hero beats)

The demo is built around three beats, in sequence, that the judges experience:

1. **The Act** — a cross-scene consequence trigger (the rescue).
2. **The Propagation** — the news visibly spreads through the NPC network.
3. **The Callback** — a distant NPC references the deed in unscripted dialogue.

### Delivery mode

**Hybrid — live demo with a scripted fallback reel.** Live interaction is primary;
a pre-recorded playthrough reel is cued as backup if the live LLM calls are slow
or fail. Standard hackathon insurance.

## 3. Foundations

Built on the **Stanford Generative Agents / Smallville** codebase
(`joonspk-research/generative_agents`, Apache-2.0, ~21.6k stars): a 2D top-down
town in **Phaser 3** (TypeScript) with a **Python FastAPI** backend, where ~25
LLM-powered NPCs each have a memory stream, reflection, daily plans, and
free-form dialogue, with information spreading via NPC-to-NPC observation.

**Procurement posture:** copy/fork the existing repo into this project as the
starting codebase — **we do not rewrite any of it from scratch.** We add the
new pieces on top and swap the LLM base URL; we do not rework what is already
there. Untouched: the agent loop
(observe → plan → reflect → dialogue), file-based memory store, Phaser
rendering, the 2D town map itself.

### LLM gateway

OpenAI-compatible SDK with the base URL swapped:

- **Primary:** MeshAPI (`https://api.meshapi.ai`) — a universal LLM gateway,
  OpenAI-compatible `ChatCompletion` API.
- **Fallback:** OpenRouter (`https://openrouter.ai`) — also OpenAI-compatible.

Both route to the same OpenAI SDK call sites; no code-shape change in the agent
loop. Fallback is automatic on primary failure.

## 5. Components & Responsibilities

### 5.1 Player Controller (frontend, TypeScript) — NEW

WASD movement and click-to-talk target selector; emits player position and
action events to the backend. Interface: `POST /player/action {type, x, y}`.

### 5.2 Director Hotspots (frontend, TypeScript) — NEW

Invisible map zones that, when the player avatar enters, trigger a scripted
in-world action (fire rescue; restaurant order). No direct backend interface —
delegates to Player Event Service via Player Controller.

### 5.3 Player Event Service (backend, Python) — NEW

Receives hotspot/player actions and canonicalizes them into structured **world
events**:

```json
{ "id": "...", "who": "player", "verb": "rescued",
  "object": "a girl", "where": "residential/fire_house",
  "when": "t0", "sentiment": "heroic" }
```

Writes to the event log and seeds the originating feed post. Interface:
`POST /events`, `GET /events?since=...`.

### 5.4 Feed Service (backend, Python) — NEW

The in-world social network as a first-class data structure. Maintains posts:

```json
{ "id": "...", "author": "...", "text": "...", "ts": "...",
  "parent": "...|null", "reach": 1, "audience": "residential" }
```

NPCs both post to and read from the feed; posts feed into agent memory as
observations via the new "feed inbox" channel. Spread mechanics (retweets /
reactions governed by audience + social proximity) produce visible propagation.
Interface: `POST /feed`, `GET /feed`, `GET /feed?npc_id=...`,
`GET /feed/stream` (SSE/WebSocket).

### 5.5 Agent Loop (backend, Python) — UNCHANGED except feed inbox

Per-NPC observe → plan → reflect → dialogue. The only change: observations now
include feed posts routed from Feed Service into the memory stream's new feed
inbox channel. No new external interface — consumes from Feed Service and the
event log internally.

### 5.6 Feed Overlay Panel (frontend, TypeScript) — NEW

Sidebar displaying the live stream of NPC posts/retweets with author handle,
reach count, and a propagation ripple animation. Subscribes to
`/feed/stream`. Pure display — no game logic.

### 5.7 LLM Client (backend, Python) — CHANGED base URL

OpenAI SDK with the base URL pointed at MeshAPI (primary) and OpenRouter
(fallback). No change to call-site shape.

### 5.8 Agent Memory Store (backend, Python) — EXTENDED

File-based store, as-is. Extended with a "feed inbox" channel so feed posts
surface as observations retrievable by relevance at dialogue time.

## 6. Data Flow — Hero Demo Timeline

Times below are demo-paced (sim fast-forward), not wall-clock.

### Beat 1 — The Act (residential district)

1. Judge walks player avatar (WASD) into the burning-building hotspot.
2. Director Hotspot fires `rescue_person` action.
3. Player Event Service canonicalizes the event:
   `{who: player, verb: rescued, object: "a girl", where:
   "residential/fire_house", when: t0, sentiment: heroic}`.
4. Event is logged **and** the first feed post is seeded unconditionally
   — authored by the rescued girl's family, audience: residential, reach: 1.

### Beat 2 — Propagation (whole town, next ~1–2 sim-minutes)

1. NPC agent loop ticks. On each observe step, NPCs pull both nearby diegetic
   dialogue (NPC-to-NPC chatter) and their feed inbox.
2. Family NPCs near the fire post and retweet. Feed Service applies spread
   rules (audience + social proximity); broader-town NPCs eventually see a
   retweet appear in their feed inbox.
3. Feed Overlay Panel streams posts live (SSE): posts ripple, reach counter
   climbs — the judges *see* the information spread.
4. Nearby NPCs also chat on the map ("did you hear about the fire?") — diegetic
   propagation reinforcing the panel.
5. Market-district NPCs (including the restaurant owner, secretly the rescued
   girl's uncle) pull the post into their feed inbox → it becomes an
   observation stored in memory tagged with the player and the event.

### Beat 3 — The Callback (market district, restaurant)

1. Judge walks player into the restaurant hotspot.
2. Director Hotspot fires `order_food`.
3. Player Event Service logs the event; then, before generating speech:
   - The Agent Loop retrieves, from the owner NPC's memory, any observation
     relevant to the rescue.
   - **YES** → dialogue action = refuse payment and reference the specific
     deed, generated by the LLM in the owner's voice, grounded in the stored
     observation (no hallucinated facts).
   - **NO** → ordinary transaction (the demo's natural graceful fallback,
     showing the system degrades cleanly).
4. The owner's line streams as a chat bubble on the map and as a post on the
   feed panel ("#OnMe today, hero eats free").

## 7. Reliability Guards

1. **Forced first post.** The rescue event unconditionally seeds the
   originating feed post — we never wait for an NPC to "decide" to post it.
2. **Deterministic reach thresholds.** By a fixed sim time, the post is
   guaranteed to have reached the market district. If natural spread is slow,
   Feed Service can "promote" the post (force audience expansion).
3. **Memory grounding.** The callback dialogue prompt receives the retrieved
   observation as fact, so the LLM references the rescue without inventing
   details.
4. **Graceful fallback.** Beat 3 degrades to an ordinary transaction if
   propagation failed or memory is missing; the demo never crashes. The
   recorded reel is cued for the crashed case.

## 8. Error Handling

- **LLM failures.** MeshAPI primary → automatic fallback to OpenRouter (same
  SDK shape). If both fail, the agent loop catches the exception, skips that
  NPC's turn, and continues; the Feed Overlay shows a warning badge so judges
  see the system as degraded rather than frozen. The recorded reel is the last
  resort.
- **Slow propagation.** Feed Service guarantees the originating post reaches
  the market district by a configured max sim time by promoting the post if
  natural spread is slow.
- **Missing memory at callback.** Dialogue retrieval falls back to a generic
  transaction line if the rescue observation is not present — the demo
  continues without the callback firing.
- **Malformed LLM output.** Structured-output validation (JSON schema) on
  events and posts; malformed responses are retried once and then dropped. No
  bad data enters the feed or memory.
- **Network egress at the venue.** OpenRouter fallback works over plain HTTPS.
  If external egress is fully blocked, the demo falls back to the recorded
  reel. Venue connectivity should be confirmed early.

## 9. Testing Strategy

Pragmatic, hackathon-bounded. Bias toward pinning the deterministic
propagation (the part that must work) with fast tests; keep LLM-dependent
assertions loose and slow-tagged.

- **Unit tests** for new backend services with mocked LLM:
  - Player Event Service canonicalization.
  - Feed Service post / retweet / spread rules.
  - "Does this post reach this NPC by time T?" — pure state function.
- **Integration test** for the propagation path: seed a rescue event → run N
  agent ticks against a stubbed LLM (canned lines) → assert the post reaches
  the restaurant owner's feed inbox within the configured max sim time and
  appears in memory. Single highest-value test; pins Beat 2 without LLM
  variance.
- **Spike test** (slow-tagged) for the callback beat: feed a real LLM a known
  grounded observation and assert the owner's line references the rescue by
  substring/pattern match. Flaky by nature; not on the critical path.
- **Manual smoke checklist** for the live demo (rescue spawns post → overlay
  streams → owner callback line appears) — run before each judging session.
- **No frontend unit tests.** Phaser/UI churn is not worth testing for a
  one-shot demo; the smoke checklist covers it.

## 10. Out of Scope

- PlayStation-native build. The demo is browser-only; no PS console port.
- 3D rendering. 2D top-down, as in base Smallville.
- Persistent player accounts, saves, or replay systems.
- NPC voice synthesis (text bubbles only for this prototype).
- Rewrites of the agent loop, memory store, or Phaser frontend stack.
- Additional hero beats beyond the three locked in Section 2.
